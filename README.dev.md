# Developer Setup

Welcome to D8-Chat. This guide walks you from a fresh laptop to a working dev
environment, then through the day-to-day workflows you'll use most: deploying
code, working with the database, and writing migrations.

If you want a higher-level tour of the architecture, the message flow, or the
blueprint layout, see `CLAUDE.md`. This file is the mechanical "how do I run
the thing" reference.

---

## What you're setting up

The primary dev environment runs the app inside a local k3s cluster managed by
DevOcho's [`auto`](https://github.com/devocho/auto) CLI. That means you get
the same container image, Postgres, Valkey (Redis), and MinIO setup you'd see
in production, without having to install any of those locally. The app
answers at `https://d8-chat.local` with a real TLS cert generated by
`mkcert`, and Python source is mounted into the pod from a PVC so most of
your edits are live without a rebuild.

If you just want a quick "run the Flask dev server directly against
SQLite/Postgres on your host," there's a non-cluster path at the bottom of
this doc. The k3s-via-`auto` path is the supported one.

## One-time setup

### 1. Install `auto`

Follow the install instructions in <https://github.com/devocho/auto>. `auto`
itself wraps Docker + k3d + `mkcert` + a handful of YAML deployment manifests
in `~/.auto/config/`.

### 2. Make sure d8-chat is registered with `auto`

The first time `auto` runs, it asks you which pods to manage. Confirm
`d8-chat` is in `~/.auto/config/local.yaml`. If you add it later, **stop and
restart the full cluster** (`auto stop && auto start`) — `auto start
<pod>` alone skips cert regeneration, which leaves Chrome showing
`NET::ERR_CERT_COMMON_NAME_INVALID` for `d8-chat.local`. This is a known
sharp edge.

### 3. Clone and configure

```bash
git clone <repo-url> d8-chat
cd d8-chat
cp example.env .env
```

Open `.env` and fill in:

- `SECRET_KEY` — generate with
  `python -c 'import secrets; print(secrets.token_urlsafe(48))'`. Must be
  ≥32 chars; the app refuses to start otherwise.
- `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` — anything you like for
  local; `auto`'s Postgres pod will pick them up.
- `MINIO_ROOT_USER`, `MINIO_ROOT_PASSWORD` — same deal.
- `OIDC_*` — leave commented unless you're testing SSO. If you set one of
  these, you must set all three (`OIDC_CLIENT_ID`, `OIDC_CLIENT_SECRET`,
  `OIDC_ISSUER_URL`); the startup validator hard-fails on partial config.
- `BRAND_*` — optional. Override `BRAND_SERVER_NAME`, `BRAND_PRIMARY_COLOR`,
  etc. if you want the mobile `/api/v1/app-config` to advertise a different
  name/color for your local environment.

The full list with descriptions lives in `example.env`.

### 4. Set up the in-tree venv (optional but convenient)

The repo ships an in-tree venv at the project root (`bin/`, `lib/`,
`pyvenv.cfg`). You don't have to activate it — call the binaries directly:

```bash
bin/python -m pytest      # runs tests against project deps
bin/pip install <pkg>     # installs into THIS venv only
bin/python smalls.py …    # see Migrations section below
```

If you'd rather use your own venv or system Python that's fine too — just
match `requirements.txt`.

## Daily loop

### Starting and stopping the cluster

```bash
auto start            # bring up the whole cluster (Postgres, MinIO, Valkey, …)
auto stop             # tear it back down
auto status           # see which pods are healthy
auto restart d8-chat  # restart only the d8-chat pod
```

`auto start` regenerates TLS certs covering `*.local` via `mkcert`. The first
run installs a local CA into your system trust store (may prompt for sudo).
If Chrome ever insists the cert is bad after a config change, `auto stop &&
auto start` is the fix.

### Deploying code changes

Most of the time you don't need to. Python source is mounted into the pod
from a PVC at `/mnt/code/d8-chat`, so editing a `.py` file under `app/`
takes effect immediately on the next request — no `auto tag` required.

You **do** need to rebuild the image when:

- `requirements.txt` changes (new pip dep)
- `Dockerfile` changes (e.g. a new apt package — we did this for `libmagic1`)
- Static asset paths change

```bash
auto tag d8-chat       # build & push image to k3d-registry.local:12345
auto restart d8-chat   # apply
```

### Tailing logs

```bash
auto logs d8-chat
```

Errors go through Python's `logging` module (no more `print()` — see the
audit history in `SECURITY_AND_ENHANCEMENTS.md`). If you set `SENTRY_DSN` in
`.env`, unhandled exceptions also flow there automatically.

## Database lifecycle

### First time on a fresh DB

```bash
auto init d8-chat     # runs init_db.py inside the pod (creates tables, admin user)
auto migrate d8-chat  # records the smalls baseline + applies any newer migrations
auto seed d8-chat     # OPTIONAL: 200 fake users + channels + messages
```

`init_db.py` creates an `admin` user. The password comes from one of:

1. `INITIAL_ADMIN_PASSWORD` env var if set (operator-controlled, deterministic).
2. Otherwise, a generated 16-byte URL-safe token, written both to stdout
   **and** to `instance/admin_credentials.txt` (mode 0600) so you have a
   recovery path if you missed the stdout output. Change the password from
   the admin UI right after first login.

### Resetting

```bash
auto stop d8-chat                    # optional but cleaner
python3 init_db.py --reset-db        # only works locally; drops + recreates
```

Or, if you want to nuke everything from inside the pod, `auto restart`
followed by `auto init d8-chat` won't drop existing tables — `init_db.py`
without `--reset-db` is idempotent. To actually drop, exec into the pod and
run `python3 init_db.py --reset-db`.

## Migrations with smalls

[smalls](https://github.com/DevOcho/smalls) is the migration tool. Files
involved:

| File | Role |
|---|---|
| `smalls.py` | Vendored CLI from the smalls repo (with the auto-generated migration template patched for Postgres). |
| `config.ini` | Tells smalls which module exposes the DB and what the current `smalls_version` is. |
| `db_bootstrap.py` | Stands up a real Peewee connection from `DATABASE_URI` *and* initializes the app's `db` Proxy so migration files can `from app.models import SomeModel`. |
| `migrations/NNNN_*.py` | The migration files themselves. |

The Flask app's `db` is a Peewee `Proxy()` that needs `create_app()` to
initialize — but smalls imports its model module at script startup, before
any Flask context exists. `db_bootstrap.py` is the bridge: when smalls does
`from db_bootstrap import db`, the bootstrap creates a real connection from
the same `DATABASE_URI` env var the app uses and binds it to the proxy at
the same time.

### Day-to-day commands

Run via `auto`, which execs smalls inside the pod where the DB is reachable:

```bash
auto migrate d8-chat       # apply all pending migrations
auto rollback d8-chat      # roll back the most recent
```

Or from inside the pod (or anywhere `postgres` resolves):

```bash
./smalls.py status                  # what's applied vs pending
./smalls.py create "add foo column" # scaffold a new migration file
./smalls.py migrate                 # apply pending
./smalls.py rollback NNNN           # roll back to version NNNN
```

Smalls connects to the DB at module import time, so commands like
`./smalls.py --help` from your laptop won't work — `postgres` doesn't
resolve there. Use `auto migrate` for routine ops.

### Writing a new migration

1. **Add the model change.** Edit `app/models.py` for new columns/tables; or
   write the SQL DDL directly inside the migration `migrate()` function.
2. **Scaffold the file:**
   ```bash
   auto exec d8-chat -- ./smalls.py create "add reset_count to users"
   ```
   Or locally, scaffold by hand — copy an existing migration as a template.
3. **Fill in `migrate()` and `rollback()`.** Use `PostgresqlMigrator` for
   column/index/constraint operations; use `db.create_tables([...])` for
   whole new tables. The auto-generated template imports the right things.
4. **Bump `smalls_version`** in `config.ini` to match the new file's number
   (e.g., `0003`). This is what `smalls magic` compares against the last
   recorded migration to decide whether to migrate up or down.
5. **Apply locally:** `auto migrate d8-chat`. Verify with `./smalls.py
   status` (inside the pod) or by inspecting `MigrationHistory`.
6. **Commit** the migration file alongside the model change. Reviewers
   should see both in the same PR.

### Baseline (`0001_initial.py`)

The first migration is a deliberate **no-op**. Existing prod databases were
created by `init_db.py`'s `db.create_tables(ALL_MODELS, safe=True)` before
smalls existed, so they already have all the original tables. The baseline
just records `0001` in `MigrationHistory` so future migrations have a clean
starting point.

For a brand-new DB, the order is `auto init` → `auto migrate`. `init_db.py`
creates the schema; `migrate` records the baseline + applies any migrations
added after the baseline (e.g. `0002_add_audit_log.py`).

## Tests

```bash
bin/python -m pytest                # full suite
bin/python -m pytest tests/test_chat.py
bin/python -m pytest -v             # verbose
bin/python -m pytest --cov=app      # with coverage
```

Tests use SQLite in-memory via `TestConfig` — no Postgres, Valkey, or MinIO
required. The `conftest.py` creates a fresh DB for each test session.
`RATELIMIT_ENABLED` and the upload-validation magic-bytes are in play, but
external services are mocked.

## Non-cluster path (rarely needed)

If you really don't want to use `auto`, you can run the Flask dev server
directly:

```bash
python3 run.py                                          # http://localhost:5001
docker compose -f docker-compose.dev.yaml up --build -d # Compose alternative
```

You'll need a Postgres reachable at the host/port your `.env` points at, a
Valkey/Redis instance, and either a MinIO instance or a stub. Most
contributors find this less convenient than just using `auto`. The k3s path
is what's tested and supported.

Things that don't work over plain HTTP:

- Session cookies have `SESSION_COOKIE_SECURE=True` — set
  `SESSION_COOKIE_SECURE=False` locally if you serve over `http://localhost`.
- Flask-WTF CSRF checks still apply; HTMX is wired to send the token via
  `X-CSRFToken` from a meta tag, so no manual setup needed.
- The WebSocket auth for `/ws/api/v1` requires the
  `Sec-WebSocket-Protocol` header (see `MOBILE_WEBSOCKET_API_DOCS.md`); the
  query-param fallback was removed.
