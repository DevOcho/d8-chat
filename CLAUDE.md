## Local Development (k3s via `auto`)

The primary local development workflow uses a local k3s cluster managed by the `auto` CLI tool (https://github.com/devocho/auto), which runs k3s inside Docker via k3d. The app is served at **`https://d8-chat.local`** with a local CA cert generated by `mkcert`.

### Cluster Lifecycle
```bash
auto start            # Start the full cluster (first run installs local CA, may prompt for sudo)
auto start d8-chat    # Start only the d8-chat pod
auto stop             # Stop the cluster
auto status           # Show cluster and pod health
auto restart d8-chat  # Restart the pod
```

### Building and Deploying Code Changes
```bash
auto tag d8-chat      # Build container image, tag it, push to local registry (k3d-registry.local:12345)
auto restart d8-chat  # Apply the new image
```

> **Note:** Python source is mounted into the pod from a PVC at `/mnt/code/d8-chat`, so Python file changes are live without rebuilding. Only `requirements.txt` or `Dockerfile` changes require `auto tag`.

### Database and Seeding (in-cluster)
```bash
auto init d8-chat     # Run init_db.py inside the pod
auto seed d8-chat     # Run seed.py inside the pod
auto migrate d8-chat  # Apply pending smalls migrations
```

### Migrations (smalls)

Schema changes after the initial deploy go through smalls (`smalls.py`,
`config.ini`, `migrations/`). The wiring:

- `db_bootstrap.py` — standalone Peewee connection used by smalls (the
  Flask app's `db` is a Proxy that needs `create_app()` first, which smalls
  doesn't run). Also initializes the proxy so migration files can
  `from app.models import SomeModel` and have it resolve.
- `config.ini` — points smalls at `db_bootstrap.db`. Bump
  `smalls_version` each time you add a migration.
- `migrations/` — `NNNN_description.py` files with `migrate()` and
  `rollback()` functions. Use `PostgresqlMigrator` (the template is patched
  for that). The first file (`0001_initial.py`) is a no-op baseline since
  existing prod DBs were already created via `init_db.py`.

```bash
./smalls.py status                  # what's applied vs pending
./smalls.py create "add foo column" # scaffold a new migration
./smalls.py migrate                 # apply pending
./smalls.py rollback NNNN           # roll back to version NNNN
```

The smalls connect happens at import time, so commands like `--help`
won't work outside a pod where `postgres` resolves. Use `auto migrate
d8-chat` for routine application.

### Push Notification Credentials (FCM / Firebase)

The pod expects a Firebase service-account JSON at `/secrets/firebase.json`,
mounted from a k8s `Secret` named **`firebase-credentials`** (see
`deployment.yaml`). The Secret is marked `optional: true`, so the pod
starts cleanly even when it doesn't exist — `push_service.init_app()`
logs `"FIREBASE_CREDENTIALS_PATH not set; push notifications disabled."`
and the rest of the app runs normally. Mobile pushes simply don't fire.

The same Secret name and mount path are used in dev and prod — only the
population mechanism differs.

**One-time setup (local dev):**
```bash
# 1. Drop the service account JSON (downloaded from Firebase console →
#    Project Settings → Service accounts → Generate new private key) at
#    secrets/firebase.json. The secrets/ folder is gitignored.

# 2. Create the k8s Secret in the local cluster:
kubectl create secret generic firebase-credentials \
  --from-file=firebase.json=secrets/firebase.json

# 3. Restart the pod to pick it up:
auto restart d8-chat

# 4. Confirm initialization in the logs:
auto logs d8-chat | grep -i firebase
#   → "Firebase Admin initialized for push notifications."
```

**To rotate the key** (e.g. after generating a new one in the Firebase
console): replace `secrets/firebase.json`, then
`kubectl delete secret firebase-credentials && kubectl create secret …`
followed by `auto restart d8-chat`.

**Production:** populate the same-named Secret via your real secret
manager (Vault / sealed-secrets / external-secrets). `deployment.yaml`
needs no changes between dev and prod.

### HTTPS / Local CA

`auto start` uses `mkcert` to generate a local CA and TLS certs covering `*.local`. The CA is installed into the system trust store automatically (sudo may be required on first run). Chrome requires the CA to be trusted at the OS level — if Chrome shows a cert error, re-run `auto start` or manually run `mkcert -install` and restart Chrome.

The `root_ca.crt` file in the project root is the local CA certificate used by the pod itself for outbound HTTPS calls (`REQUESTS_CA_BUNDLE=./root_ca.crt` in `deployment.yaml`).

**Known gotcha:** `auto start <pod>` skips cert regeneration. If you add a new pod to `~/.auto/config/local.yaml` and start it individually, Chrome will show `NET::ERR_CERT_COMMON_NAME_INVALID` because the pod's `<name>.local` domain won't be in the cert SANs. Fix: stop and restart the full cluster with `auto stop && auto start` so certs are regenerated with all current pod domains included.

## Architecture

**D8-Chat** is a self-hosted Slack/Teams alternative. The stack:
- **Backend**: Flask 3 + Peewee ORM, Gunicorn in production
- **Database**: PostgreSQL (production), SQLite in-memory (tests via `TestConfig`)
- **Real-time**: WebSockets (`flask-sock`) + Redis/Valkey Pub/Sub
- **File Storage**: MinIO (S3-compatible)
- **Frontend**: Server-rendered Jinja2 + HTMX + Bootstrap 5 — no SPA framework
- **Auth**: bcrypt login, optional OIDC SSO (Authlib), API tokens for mobile

### Message Flow

The WebSocket is **push-only** (server → client). Both web and mobile clients
**send** messages over HTTP; the socket only delivers realtime updates.

1. Client POSTs the message:
   - Web: `POST /chat/conversations/<conv_id_str>/messages` (session auth); the
     compose form is `hx-post` (see `chat_input_*.html`). The sender gets a real
     success/failure instead of a fire-and-forget WS frame.
   - Mobile: `POST /api/v1/conversations/<conv_id_str>/messages` (Bearer token).
   Both funnel through `routes.handle_inbound_message()` (shared with the
   legacy WS `send_message` path still accepted for back-compat).
2. `chat_service.handle_new_message()` creates the DB record, extracts mentions/hashtags/polls
3. `chat_manager.broadcast(conversation_id, payload)` publishes to Redis `chat:{conversation_id}`
4. The supervised background listener in `ChatManager.listen_for_messages()`
   receives from Redis and pushes to connected WebSockets (reconnects with
   backoff if Redis drops — it never silently dies)
5. HTMX on the client applies the rendered HTML fragment via OOB swap

Notification fan-out (badges, sounds, desktop + FCM push) runs off the hot path
via `background.spawn_background`. After a WebSocket reconnect the web client
backfills anything missed while disconnected via
`GET /chat/conversations/<id>/messages/since/<last_id>` (pub/sub is
at-most-once, so gaps are the client's job to fill).

### Blueprints (`app/blueprints/`)

| Blueprint | Prefix | Purpose |
|-----------|--------|---------|
| `main_bp` | `/` | Chat interface, WebSocket handler (`/ws`) |
| `auth_bp` | `/` | Login, logout, OIDC callback |
| `channels_bp` | `/channels` | Channel CRUD and membership |
| `dms_bp` | `/dms` | Direct messaging |
| `messages_bp` | `/messages` | Edit, delete, reactions |
| `files_bp` | `/files` | Upload/download |
| `admin_bp` | `/admin` | User and workspace management |
| `polls_bp` | `/polls` | Poll creation and voting |
| `api_v1_bp` | `/api/v1` | REST + WebSocket API for mobile clients |

### Key Modules

- `app/__init__.py` — App factory (`create_app()`); registers blueprints, starts ChatManager thread
- `app/models.py` — All Peewee models; `BaseModel` adds `created_at`/`updated_at`
- `app/chat_manager.py` — `ChatManager` class: per-user multi-socket registry, supervised Redis pub/sub listener, and TTL-based cluster presence (a `presence:online` Redis ZSET scored by heartbeat, so a crashed worker's users age out instead of being pinned "online")
- `app/ws_utils.py` — WebSocket send-lock / `LockedSocket` hardening; `app/background.py` — `spawn_background` for off-hot-path work
- `GET /healthz` — realtime-aware readiness check (Redis ping + listener heartbeat age + trivial DB query); wired to the k8s readiness probe, not liveness
- `app/chat_service.py` — Business logic for message creation, mentions, notifications
- `app/sso.py` — OIDC SSO integration (Authlib)
- `config.py` — `Config` (production) and `TestConfig` (SQLite, no Redis) classes

### Conversation IDs

Conversations are identified by string keys: `channel_{channel_id}` or `dm_{user1_id}_{user2_id}`. These keys are used as Redis pub/sub channel names and stored on the `Conversation` model as `conversation_id_str`.

### WebSocket (Redis) Message Topics

- `chat:{conversation_id}` — Broadcast to all clients in that conversation
- `user:{user_id}` — Send to a specific user (DM notifications, presence events)
- `global:*` — Broadcast to all connected clients (workspace-wide events)

### Mobile API Auth

Mobile clients use signed Bearer tokens: `Authorization: Bearer d8_sec_<token>`. Tokens are 30-day signed JWTs generated by `generate_api_token(user_id)` and validated by the `@api_token_required` decorator in `api_v1.py`.

## Environment Setup

Copy `example.env` to `.env`. Required variables:
- `SECRET_KEY` — Flask secret
- `DATABASE_URI` or `POSTGRES_*` components
- `VALKEY_URL` — Redis/Valkey connection string
- `MINIO_ENDPOINT`, `MINIO_ROOT_USER`, `MINIO_ROOT_PASSWORD`, `MINIO_BUCKET_NAME`, `MINIO_PUBLIC_URL`

Optional SSO: `OIDC_CLIENT_ID`, `OIDC_CLIENT_SECRET`, `OIDC_ISSUER_URL`

## Testing Notes

Tests use SQLite in-memory via `TestConfig` — no Redis or MinIO required. The `conftest.py` creates a fresh DB for each test session. When adding tests for new features, follow the pattern in `tests/test_chat.py` and `tests/test_api_v1.py`.
